# Domain — Example (a model, not a live domain)

> **This is a filled example to show what a good capsule looks like.** It is not a live domain (it is not in `_registry.md`). Read it, then delete it or adapt it into your own. Your real domains are the `work` / `side-project` starter capsules.
>
> **Entry point:** read this file, then the "Recent state" at the bottom.

## Where it lives
This domain's work is a code project in an **external repo**: `~/Sites/my-app`. When operating in this domain, work in that repo — it has its own context and rules. The command center routes you here; the code lives there.

A domain doesn't have to be code. A "research" or "writing" domain might instead capture notes and analysis in an **external notes store**, e.g. `~/notes/research/`. When operating such a domain, read and write there.

## Rules
- Follow the conventions of wherever the work lives (the repo's own rules, or the notes store's structure).
- Keep this domain's context here; don't leak it into other domains.

## Permissions
- Default: write-with-confirmation for changes; the safety gate (plan → review → approve → execute) always applies.

## Recent state
- Scaffolded the command center and read the guide. (The agent appends or links recent work here so the next session has context.)
